# Screen & Camera Recorder — 절대 조건 (개발 원칙)
> 최종 정리본 · SWC 재설계 기준 문서

---

## 1. 안정성

**1-1. 무한 루프 금지 / 불필요한 루프 제거**
모든 루프에는 반드시 종료 조건이 있어야 하며, 루프가 실제로 필요한지 먼저 검토할 것.

**1-2. 근원 원인 우선 수정**
문제 발생 시 근원 원인 분석 → 원인 코드 수정 우선.
영향 범위가 커서 직접 수정이 어려울 때만 변수를 temp에 할당해 독립 스레드로 격리하는 등 우회 방법을 적용.
우회 적용 시 반드시 주석으로 근원 원인과 우회 이유를 명시할 것.

**1-3. 방어적 코딩 — 데이터 포트 변수화**
기능 모듈에서 처리되는 데이터를 변수로 만들어 해당 변수가 기능의 포트 역할을 하도록 설계.
타 기능이 그 데이터를 시그널·메서드를 통해 컨트롤 가능하게 할 것.
과도한 변수화는 메모리 증가와 성능 저하를 유발하므로, 임베디드의 공용 메모리 컨셉
(구조체 / 공용체 / 비트필드)을 참고해 꼭 필요한 곳에만 적용할 것.

**1-4. 병목 방지 — 버퍼 + 스케줄링**
병목 발생 여지가 있는 경로(프레임 캡처, CAN RX, 로그 출력 등)에
버퍼 + 스케줄링을 두어 프로세스 중단 방지.
버퍼 크기는 `maxsize` 또는 `maxlen`으로 반드시 상한을 설정할 것 (무제한 누적 금지).

**1-5. Silent Failure 금지**
`except Exception: pass` 형태의 예외 묵살 금지.
예외는 최소 `signals.status_message.emit(str(e))` 또는 로그 파일에 기록할 것.
정말 무시해도 되는 예외는 반드시 주석으로 이유를 명시할 것.

**1-6. 중복 초기화 금지**
동일한 객체(타이머, 스레드, DB 연결 등)를 같은 스코프에서 두 번 이상 초기화 금지.
두 번째 초기화가 첫 번째를 덮어쓰면 첫 번째 리소스가 해제되지 않고 누수됨.
`stop()` / `close()` 정리 코드도 중복 호출 없이 단 한 곳에서만 실행할 것.

**1-7. 자원 해제 순서 명시 (Cleanup Order)**
프로그램 종료(`closeEvent`) 또는 SWC 재시작 시 스레드·장치·DB 커넥션을
의존성의 역순으로 해제할 것.
해제 순서가 틀리면 Deadlock 또는 프로세스 좀비가 발생함.
권장 해제 순서: `UI 패널 → 커널 → CAN → 카메라/캡처 엔진 → DB → 전원 컨트롤러`

**1-8. 물리적 예외 복구 (장치 / 디스크)**
장기 실행 프로그램 특성상 USB 카메라 단선, 해상도 변경, 디스크 용량 고갈은 반드시 발생함.
- **카메라 연결 끊김**: 일정 횟수 재연결 시도(Retry) 후 실패 시 사용자에게 알림. 재시도 루프는 반드시 종료 조건과 sleep을 가질 것.
- **디스크 용량 95% 이상**: 즉시 녹화를 안전하게 중단하고 자원을 해제한 후 사용자에게 알림.
- **디스크 쓰기 실패**: 예외를 캐치해 현재 세그먼트를 닫고 로그에 기록. 프로세스 전체를 죽이지 말 것.

## 2. 아키텍처

**2-1. 로직 캡슐화 — 기능 단위 독립성**
기능 추가·수정 시 전체 코드에 영향이 없도록 로직을 철저히 캡슐화할 것.
기능(= 함수)의 변수화를 철저히 하는 것과 같은 의미.

**2-2. 기능별 객체 구현 — 다수 인스턴스 고려**
하나의 기능에서 다수의 인스턴스가 생성될 수 있는 가능성까지 고려하여 설계.

**2-3. API Doc 상시 동기화**
새 기능 생기면 설정 함수 + 실행 함수 모두 `_API_DOC` 및 `_SYS_STATIC`에 즉시 업데이트.
코드와 Doc이 불일치하면 커널·AI Chat이 잘못된 함수를 호출하는 원인이 됨.

**2-4. 스레드 안전 원칙 — 양방향**
- 백그라운드 스레드 → UI: `pyqtSignal` 또는 `QMetaObject.invokeMethod(QueuedConnection)`만 사용. `setText`, `setChecked`, `appendPlainText` 등 직접 호출 금지.
- UI 메인 스레드 → 백그라운드: 무거운 연산(I/O, 영상처리, CAN 통신 등)을 메인 스레드에서 직접 호출 금지. 작업 요청은 Queue 또는 시그널로 전달하고, 완료는 시그널로 수신.

**2-5. 전역 변수 최소화 — 의존성 주입 (DI)**
`BASE_DIR`, `DB_PATH` 같은 상수는 `common/constants.py`로 분리.
`_can_manager` 같은 싱글턴 인스턴스는 `MainWindow`에서 생성 후 각 컴포넌트에 주입.
모듈 전역에 인스턴스를 선언하면 SWC 분리 시 순환 임포트의 핵심 원인이 됨.

**2-6. 타입 어노테이션 필수**
Public 메서드 및 SWC 경계를 넘나드는 함수에는 파라미터·반환 타입 어노테이션 필수.
타입 없이는 SWC 간 인터페이스를 파악하기 어렵고 유지보수 비용이 급증함.

**2-7. 매직 넘버·매직 스트링 금지**
UI 색상값, 경로명, 타이머 주기, FPS 상수 등 하드코딩된 값은
`constants.py` 또는 각 SWC 상단의 상수 블록으로 분리할 것.

**2-8. 설정 저장·로드 단일 책임**
`_load_settings` / `_save_settings`는 여러 클래스에 분산하지 말고
단일 책임 클래스(`SettingsManager`)에서 일원화할 것.
분산 시 키 누락이 반복 발생하는 구조적 원인이 됨.

## 3. SWC 분리 원칙

**3-1. 단방향 의존 그래프 유지**
```
common → engine_swc → (recording_swc / kernel_swc / can_swc) → ui_swc → main_window
```
역방향 임포트 발생 시 시그널 또는 콜백으로 분리.

**3-2. SWC 간 데이터 교환 규칙**
SWC가 다른 SWC의 내부 속성(`_xxx`)에 직접 접근 금지.
필요한 데이터는 public 메서드나 `pyqtSignal`로만 노출.

**3-3. common 비대화(Fat Common) 방지**
각 SWC 전용 상수는 해당 SWC 내부에 둘 것.
common에는 프로그램 전역에서 공유하는 핵심 규격(경로 상수, Signals 정의, 공용 데이터 모델)만 위치시킬 것.
common이 모든 것을 알아야 하는 구조가 되면 SWC 분리 의미가 없어짐.

**3-4. 설정 키 Prefix 소유권**
각 SWC는 자신의 prefix 키만 저장·로드.
(예: `can_swc` → `can_` prefix, `kernel_swc` → `kernel_` prefix)
타 SWC의 키에 접근 금지. 키 충돌과 누락을 구조적으로 방지.

**3-5. PyInstaller spec 파일 관리**
새 SWC 패키지·리소스 추가 시 `collect_submodules()` / `datas` 업데이트 필수.
빌드 후 오프라인 PC에서 반드시 동작 검증.

## 4. 배포

**4-1. PyInstaller EXE 빌드**
타 PC에서 별도 설치 없이 실행 가능하도록 소스 구현.
보안 정책으로 EXE 생성 시 파일이 삭제되는 PC 환경 대응:
- 1차 BAT (인터넷 가능 PC): 의존성 수집
- 2차 BAT (오프라인 PC): EXE 생성

**4-2. 리소스 경로 변환 함수 필수 (`resource_path()`)**
PyInstaller EXE 실행 시 리소스는 임시 폴더(`sys._MEIPASS`)에 풀림.
소스코드 내 모든 파일·리소스 경로 참조는 반드시 `resource_path()` 공통 함수를 거칠 것.
단순 상대 경로(`./icon.png`) 사용 시 EXE 환경에서 크래시 발생.

```python
def resource_path(relative: str) -> str:
    base = getattr(sys, '_MEIPASS', os.path.dirname(os.path.abspath(__file__)))
    return os.path.join(base, relative)
```

**4-3. 아두이노 코드 연동**
플랫폼 코드 변경으로 아두이노 코드 변경이 필요한 경우 두 파일을 함께 수정·제공.

## 5. DB 동기화

**5-1. 초기 실행 Default 처리**
초기 실행 시 DB 없으면 default 값 사용. 이후 패널 설정값과 DB 항상 동기화.

**5-2. Debounce 2초 — 모든 설정값 변경에 적용**
모든 기능의 패널 설정값 변경 시 DB 저장 로직 반영.
휠 드래그·토글처럼 빠르게 연속 변경되는 경우 프로세스 과부화 방지를 위해
값 변경 시점부터 2초 지연(debounce) 후 저장.
(2초 이내 추가 변경 시 타이머 리셋 → 마지막 변경 후 2초 뒤 1회만 저장)

**5-3. DB 접근 직렬화 — 동시성 제어**
SQLite는 동시 쓰기 시 `database is locked` 오류 발생.
모든 DB 읽기/쓰기는 `SettingsManager` 단일 클래스 또는 전용 DB 큐를 통해
순차적으로(직렬화) 처리할 것. 다수 스레드의 직접 DB 접근 금지.

**5-4. DB 설정값 기준 동작**
DB에 저장된 설정값을 기준으로 커널 / AI Chat / 외부 통신 모두 동작.

`함수명(파라미터1, 파라미터2, ...)` 형태로 호출 가능하도록 구현.

**6-2. API Doc 모달 버튼**
Control Panel 최상단에 기능별 함수 Doc 모달 버튼 배치.

**6-3. AI Chat 스크립트 생성 능력**

`"요청을 ~로 이해해 이렇게 작성했습니다. 맞을까요?"` 형식으로 되물을 것.

## 8. 로깅

**8-1. `print()` 대신 구조화된 로그**
EXE 배포 환경에서는 콘솔이 없어 `print()`가 소실됨.
Python `logging` 모듈로 교체. 레벨(`DEBUG / INFO / WARNING / ERROR`)별 분리.
파일 핸들러 → `BASE_DIR/app.log`에 기록.

**8-2. Rotating File Handler**
장기 실행 환경에서 로그 파일 무한 증가 방지.
`RotatingFileHandler(maxBytes=5MB, backupCount=3)` 적용.

## 9. 저장 경로 및 파일 관리

**9-1. 경로 설정 패널 기준 저장**
녹화 / 수동녹화 / 캡처 / 블랙아웃 등 모든 영상·데이터 저장은
경로 설정 패널에서 지정한 경로에 저장.

**9-2. 커널 로그·결과 파일 폴더 트리 반영**
커널 로그 및 결과 파일도 경로 설정 패널 기준 폴더 트리
(`vehicle_type / YYYYMMDD / TC-ID / extra_segments`) 하위에 저장.

**9-3. 디스크 용량 자동 관리 (FIFO 정책)**
저장 경로의 남은 용량이 설정 임계값 이하이거나,
보존 기간(기본 30일)이 지난 녹화 파일·로그는 자동 삭제(FIFO) 옵션 제공.
자동 삭제 기능은 기본 OFF이며, 사용자가 명시적으로 활성화해야 동작.
삭제 전 반드시 로그에 기록할 것.

## 10. 테스트·품질

**10-1. SWC별 단위 테스트**
각 SWC는 UI 없이 로직만 검증하는 단위 테스트를 최소 1개 이상 작성.
`pytest`로 실행 가능하도록 구성.

**10-2. 인터페이스 변경 시 호출부 동시 수정**
SWC 간 인터페이스(시그널 시그니처, public 메서드)가 바뀌면
해당 인터페이스를 사용하는 모든 호출부를 함께 수정할 것.
인터페이스 변경 후 호출부 방치 시 런타임 크래시 발생.
